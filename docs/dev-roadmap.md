# AccelFury `af` Quality And Functional Maturity Roadmap

Date: 2026-05-24

Scope: current implementation status against the expanded MVP-0/MVP-1
specification, with the next engineering milestones for quality, usefulness and
functional maturity.

## Current Compliance Snapshot

Status: **Alpha readiness target: MVP-0/MVP-1 baseline operational; MVP-2+
surfaces staged**.

The alpha-supported CLI surface is intentionally limited to the
manifest-first development loop: `af doctor`, `af self check`,
`af manifest validate`, `af core check`, `af core lint`, `af core sim`,
`af core report`, `af wrapper generate`, and `af ci generate`. Other command
families can be useful, but their flags, report details, and backend coverage
remain staged until a later readiness gate promotes them.

Implemented and verified in the codebase:

- Rust workspace with CLI, manifest, RTL inspector, backend, report, security,
  board DB, wrapper and CI crates.
- `af doctor`, `af manifest validate/migrate`, `af core check/lint/sim/formal/package/report`,
  `af board list/check/matrix`, `af build`, `af flash`, `af backend list/run`,
  `af wrapper generate`, and `af ci generate`.
- Manifest-first validation for source files, include dirs, testbenches, top
  module text, ports, clocks and resets.
- v0.3 optional semantic contracts for FIFO/reset/generic protocol behavior,
  manifest-vs-RTL parameterized port width checks, same-workspace dependency
  path resolution, stream-FIFO adapter hints, generic protocol adapter hints,
  and generated ready/valid FIFO wrappers.
- Verilator lint/smoke path with structured `BackendUnavailable`.
- FuseSoC `.core` export.
- LiteX wrapper skeleton/reference dry-run.
- Yosys syntax/synthesis smoke backend path.
- JSON and Markdown report generation with tool versions, commands, artifacts,
  warnings and limitations.
- Tang Nano 20K and Tang Primer 20K board manifests with unverified resources
  explicitly marked.
- Docker open-source runtime for Verilator/FuseSoC/LiteX/Yosys, SMT solver
  debugging and CI smoke.

Still intentionally out of scope:

- full SystemVerilog parser/signoff;
- not timing closure, CDC/RDC signoff or vendor bitstream production;
- GUI, package registry and automatic generation of complex hardware logic;
- production Vivado/Quartus/Radiant flows;
- PDM-to-PCM conversion in `af-pdm-rx`.

## Roadmap

### M0: Reliability Hardening

Goal: make every existing command deterministic, structured and non-panicking.

Work:

- keep report aliases stable: legacy `core-check.json` plus spec-style
  `core_check_report.json`, `lint_report.json`, `simulation_report.json` and
  `build_report.json`;
- expand broken fixtures for all CLI subcommands, not only `core check`;
- add command-duration and stdout/stderr log-file artifacts;
- enforce redaction before reports serialize command output;
- add snapshot tests for doctor, lint, sim, package and build reports.

Exit criteria:

- invalid input never panics;
- every failure has code, message, hint and exit code;
- all generated files carry `Generated by AccelFury IP Toolchain`.

### M1: Docker-First Open-Source Flow

Goal: remove external host blockers for normal development.

Work:

- keep `Dockerfile`, `docker-compose.yml`, devcontainer and CI Docker job as
  the canonical OSS runtime;
- pin or lock Python packages for FuseSoC/LiteX when release reproducibility
  matters;
- add Docker smoke artifacts to CI uploads;
- add a local `make smoke` wrapper for the Docker flow.

Exit criteria:

- clean checkout can run Verilator simulation, FuseSoC export, LiteX skeleton
  generation and Yosys checks in Docker;
- host without FPGA tools still gets structured `BackendUnavailable` outside
  Docker.

### M2: Backend Maturity

Goal: make backends plug in without touching `af-cli`.

Work:

- move backend dispatch into a registry/planner service;
- persist prepared `CommandSpec` plans before execution;
- implement `af backend run` replay from a saved plan;
- add Yosys report metrics and expected-artifact checks;
- keep SymbiYosys/openFPGALoader/vendor backends staged until artifacts exist.

Exit criteria:

- new backend crates register capabilities through one interface;
- all backend commands are logged as `program + args[]`, never shell strings.

### M3: Report Contracts

Goal: align reports with the SRS JSON contracts.

Work:

- introduce typed `simulation_report`, `lint_report`, `build_report` and
  `package_report` structs;
- add `started_at`, `duration_ms`, reproducibility, host OS and environment
  hash fields;
- write stdout/stderr logs as artifacts and reference them from reports;
- add resource/timing summaries as explicit `available=false` blocks until real
  vendor reports are parsed.

Exit criteria:

- reports are both human-readable and machine-readable;
- false confidence is prevented by mandatory limitations.

### M4: LiteX Reference Design

Goal: move from wrapper skeleton to runnable reference integration.

Work:

- generate a minimal LiteX top that instantiates handwritten RTL only;
- add syntax checks for generated Python;
- capture LiteX command plan and logs;
- keep Tang Primer 20K experimental until constraints are bench-validated.

Exit criteria:

- generated wrapper is syntactically stable;
- no generated CDC/FIFO/audio filter/bus bridge appears silently.

### M5: Vendor Orchestration Boundary

Goal: support local/vendor runners without shipping vendor tools.

Work:

- add Gowin project/Tcl generation as the first vendor path;
- detect `gw_sh`/`programmer_cli` and record version/EULA warnings;
- parse resource/timing summaries only from existing vendor reports;
- keep Vivado/Quartus/Radiant/Diamond as script-generation or future backends.

Exit criteria:

- vendor commands are optional, no-shell and fully logged;
- CI remains green without vendor tools.

## Universal-Core Roadmap (registry-driven)

The universal-core registry (`registries/cores.registry.json`) tracks which
`af_*` cores AccelFury commits to. The lineage below is a roadmap, not a
delivery commitment, and the only authoritative status is what
`af core registry list` returns.

### Field arithmetic lineage

The `af_mod_add` example seeds a longer field-arithmetic line that the
manifesto explicitly names as a commercially-relevant target:

`af_mod_add` → `af_mod_sub` → `af_mod_mul` → `af_mod_reduce` →
`af_ntt` → `af_msm` → `af_poseidon` → `af_merkle` → PLONK-like building blocks.

Each step keeps the same v0.3 manifest shape, adds `portability_level`,
`priority`, `maturity`, and at least one entry under
`[[verification_required]]`. Promotion from `experimental` to `preview` to
`beta`/`stable` is gated by `af core report` evidence rows, not by
narrative commits.

The first six lineage cores (`af_mod_sub`, `af_mod_mul`, `af_mod_reduce`,
`af_ntt`, `af_msm`, `af_poseidon`) are registered as tracked-only entries in
[`registries/cores.registry.json`](../registries/cores.registry.json) with
`tracking_issue = "manifesto-roadmap: tracked-only, no in-tree RTL yet"`. Use:

```bash
af core registry list --json | jq '.cores[] | select(.category == "field_arithmetic" or .category == "ntt_fft" or .category == "hash" or .category == "msm_toy")'
```

to inspect them. They surface in `af core registry list` with priority/portability
defaults; promote them to in-tree examples (with manifest + RTL + tb) as RTL
work lands.

### Stream / memory / peripheral lineage

The P0/P1 cores listed in `registries/cores.registry.json` cover the
non-arithmetic core set: `af_stream_skid`, `af_stream_pipeline`,
`af_sync_fifo`, `af_async_fifo`, `af_ram_sdp`, `af_reset_sync`,
`af_pulse_sync`, `af_cdc_handshake`, `af_gpio`, `af_timer`, `af_watchdog`,
`af_uart`, `af_spi_master`. `af_reset_sync` is the first one wired up as a
standalone in-tree example (`examples/af-reset-sync/`).
